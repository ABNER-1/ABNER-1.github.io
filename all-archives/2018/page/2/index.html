关于SSM框架的一些理解

05 7月 2018

SSM框架

最近一段时间都在接触SSM框架，之前可能对其理解不到位，有了很多错误的理解。这次实训也的的确确学到了不少。

SSM 由 Spring MVC + Spring boot + Mybatis三部分组成。

  • Spring MVC负责view层的显示
  • Spring boot负责业务模型
  • Mybatis负责与数据库的交互

值得一提的是maven，maven是搭建框架的底层，我的理解是为项目引入种种包，若不适用maven进行配置，可以参考写代码时手动引入依赖包的过程。具体实现方面不算是特别了解，应该和gradle差不多。

maven中最重要的是pom.xml文件，里面定义了整个项目的maven配置，其中包括版本、jdk版本、tomcat版本以及包依赖等等。

web项目中重要的配置文件是web.xml，其中定义了spring的配置、springMVC的配置以及前端到controller层的字符类型（若不进行配置，默认的编码会导致中文乱码）

通常web.xml中几个文件会单独写一个xml文件方便配置。
如 spring-mvc.xml文件中包含的是springmvc的配置，包含了controller层的扫描位置（方便url解析），以及controller层中跳转的string处理，例如加前缀后缀等。

applicationContext.xml包含了spring中各种配置，包含各层的扫描位置，数据库配置文件的引入，数据库处理类型的声明等。

更多的log4j等配置暂时没有了解的更详细。

关于在maven目录中引入的kaptcha验证码组件等、以及如何在ssm项目中引入开源项目库等还需要更多的练习去学习。

AES算法模式
CBC CTR OFB CFB GCM XTS ECB

CBC加密部分 （密码分组链接）
    思想： 明文块与前一个密文块 进行异或操作
    第一个明文块和初始化向量异或，通过加密密钥加密得到密文块1
    第i个明文块和第i-1个密文块异或操作，然后通过加密密钥得到密文块i （i > 1）
解密过程
    密文块1先用加密密钥进行解密，再通过和初始化向量进行异或得到明文块1
    密文块i 用加密密钥进行解密， 再通过异或密文块i-1  得到明文块i

CTR算法模式 （计数器模式）

    加密过程
        （初始化文本+counter）用key加密，异或明文，得到密文
            counter++;
    解密过程
        （初始化文本+counter）用key加密，异或密文，得到明文
            counter++;


异同：CBC需要填充，CTR不需要填充
    CTR任意block不需要与前面的链接，适用流式处理的加密场景

jre中规定 如果被写对象的类型是String，或数组，或Enum，或Serializable，那么就可以对该对象进行序列化，否则将抛出NotSerializableException。

隐私数据不希望被序列化

使用默认序列化不符合要求，重写toString（）方法，或者使用transient关键字
